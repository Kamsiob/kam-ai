## Phase 1 · The working core

Goal: a person could use this daily. Everything in this phase is specified visually in DESIGN.md; implement to match, including motion, staggers, swipe behaviors, scroll edge fades, back navigation direction, and the brand bar with back arrow.

Model tiers and downloads. Three tiers, all one model family: Gemma 4, chosen because its size range covers the whole envelope under one licence (Apache 2.0, no asterisk on any tier) and one prompt format (the Gemma `<start_of_turn>` layout). The lineup, verified against the live Hugging Face API and hash-checked on device: Basic for 8 GB RAM and up is Gemma 4 E2B (about 3.1 GB at Q4_K_M); Balanced for 12 GB and up is Gemma 4 E4B (about 5.0 GB); Best Available for 16 GB and up is Gemma 4 12B (about 7.1 GB). There is no Qwen anywhere. Read total device RAM locally (this never leaves the device and needs no Data Safety disclosure), grey out tiers the device cannot hold, and recommend the tier that leaves comfortable headroom rather than the maximum. Show download size before any download. Downloads show progress, survive interruption with resume or clean retry (pausable and resumable, running in the background), verify a known hash before activation, and store into app-managed storage. Each model declares its capabilities (text, document attachments, and, where a model supports them, images or voice) as data in the catalog, so the interface adapts without hardcoding by model name; the ChatFormat enum keeps the per-model prompt design so a future family drops in cleanly. Multi-model management (an Advanced section listing other compatible models, switching the active model while keeping every conversation, and safe deletion of the active model with fallback to another) was added in the PART 2 update.

Onboarding. Five slides exactly as written in DESIGN.md section 9, opaque themed background, swipeable, tappable dots, skip link that vanishes on the last slide, replayable later from Settings. The tier slide performs the real RAM detection and starts the real download.

Conversations and modes. There are four sibling modes: General (everyday questions, formerly named Chat), Logic Partner (argues the other side), Brainstorm (pulls ideas out of you instead of handing them over, see below), and Workbench (Bench, the paste-and-transform surface). The Chats bottom-nav tab keeps its name because it holds every conversation regardless of mode. Each mode has a fixed system prompt re-injected every turn; Brainstorm's is written as an ordered method-selection checklist (Part 1 of the Four-Mode Update, DECISIONS.md). Mode is per conversation, and a conversation records every mode it has used (modesUsed) for the chat-row dots and the mode filter. The interface for the four modes (the segmented mode control that replaces the New chat button, the in-conversation mode picker with per-mode banners and midstream notices, the per-mode empty-state nudges, the chat-row mode dots and mode filter) is landing across issues #26 to #30 and is described in DESIGN.md as it ships. Streaming responses with a typing indicator, stop generation, regenerate that replaces, edit that truncates and re-answers, no branching. Input is disabled while a response streams. Auto-titled conversations, through one shared titling path so a conversation gets a good title regardless of the entry point it was created from (fixed in the refinement pass, issue #17). Assistant responses render Markdown (headings, bold and italic, lists, inline code and code blocks, links). The open conversation has a titled header with an overflow menu carrying Rename, Move to project or Remove from project, Archive, and Delete; archived conversations live in a separate Archived view (issue #20). Action row under every AI response: bookmark (the save-to-Follow-ups action, an amber filled bookmark when set, replacing the earlier flag icon, issue #9 and DECISIONS.md "The bookmark, not the flag"), copy, share, play (wired to a stub until Phase 2, hidden if no TTS voice is installed, toggling to stop while speaking), regenerate, and an overflow with Report a response, Share thread, and Export. Mode is stored per conversation and persists across reopening; switching mode drops a quiet centered system note into the transcript at the switch point, and Logic Partner shows a persistent non-amber banner while active (issue #12).

System prompts and guardrails. Maintain fixed, non-editable system instructions per mode, re-injected with every request because small models drift. All modes share the hard rules: never sycophantic, no flattery, no agreeing to be agreeable, challenge weak reasoning; no characters, no roleplay, no pretending to be a person or companion, no emotional manipulation, no getting personal; refuse sexual content and anything facilitating illegal activity, on top of the base model's own safety training. Logic Partner's instructions define a full method, not just an attitude. It opens by restating the user's position in one line so both sides know what is being tested, then challenges it: unstated assumptions, internal contradictions, feasibility gaps, unpriced tradeoffs, second-order effects, and the strongest version of the opposing case, using pointed questions where a question cuts deeper than a statement. Disagreement must be earned, never performed: when the user's point is sound it concedes plainly and moves to the next weak spot, because reflexive contrarianism is sycophancy inverted. It does not fold under mere pushback; it changes position only in response to actual new reasoning, and says what changed its mind when it does. It attacks the idea, never the person: no sarcasm, no dominance games, plain and even, and it is the same voice as the rest of the app, never a debate character or persona. Because small models hallucinate recalled facts, it argues primarily from the user's own stated premises, logic, consistency, and tradeoffs rather than from factual recall; when an argument turns on a fact, it says so explicitly and points at saving it to Follow-ups instead of inventing evidence. If the user brings emotional distress rather than an idea to test, it declines to grind, says plainly that this is not a debate topic, and suggests Chat mode. On request or as a thread winds down, it can summarize where the idea stands: strongest objections raised, what would change its assessment, and what is worth verifying. Advanced model settings (temperature, top-p, and similar) are never exposed anywhere in the UI, not even behind a toggle; choose sensible fixed values internally.

Memory, Projects, and instructions. Projects are a bottom-nav tab (added in the refinement pass, issue #2): a named container with persistent instructions injected into every chat inside it, plus its own conversation list, with chats movable in and out. Project instructions are exclusive to that project and never leak into system-wide instructions or into chats outside the project. A lightweight memory system stores durable user facts the model can be given within each tier's context budget; memory must be transparent: a screen listing exactly what is remembered, with per-entry delete. Retrieval is real relevance (keyword plus recency, prefix-matched) within a small slice of the context budget, and facts are extracted in batches with de-duplication (issue #16; contradiction supersession and a "memory influenced this" indicator remain open). Memory quality varies by tier and that is acceptable. Separately, Settings holds system-wide Custom instructions the user writes once that apply to every chat including inside projects (issue #15). The prompt precedence, fixed and documented in DECISIONS.md, is: the hard and per-mode rules always win, then the user's system-wide instructions, then the project's instructions, then memory.

History and navigation. A first-class Chats screen per DESIGN.md: three switchable views (comfortable, compact, grid), with compact as the default, compact rows carrying no leading letter icon, and the last-used view remembered and restored on every launch, collapsible Pinned section, Recent list, swipe actions for pin, archive, delete at full row height (long-press menu in grid view; the earlier clipped rail was fixed, issues #4 and #19), and search across all conversations (keyword search now; leave a clean seam for semantic search later via an on-device embedding model, and note it in Being considered). Bottom navigation is Projects, Chats, Follow-ups, Discover (the earlier standalone New nav item was removed in the refinement pass to make room for Projects, issue #2); starting a new chat now lives on the Chats screen itself via the segmented mode control. There is no Today tab: it was considered and deliberately cancelled (Four-Mode Update Part 9, issue #37, DECISIONS.md), so the navigation stays these four.

Follow-ups. The single destination for everything saved anywhere in the app. Bookmark any AI response, or a selected portion of one, into the Follow-ups tab: snippet, link back to the source conversation, optional note, optional project link, complete and incomplete states, completed group collapsed by default, swipe to remove completed items. A source filter row (All, Chat, Logic, Discover) narrows the list (issue #10). Saving is unified: a bookmarked Discover moment lands in this same list with a Discover source chip and reopens as a grounded discussion when tapped, and the Discover page's own Saved section is a filtered view of this list rather than a separate store (issue #9, DECISIONS.md "Unified saving"). This feature is deliberately simple, not a task manager.

Settings, About, Q and A, and information screens. Implement the grouped Settings. On this device: Model (with the capability chips and Advanced list described above), Voice (stub until Phase 2), Storage listing every downloaded artifact with delete. Personalization: Appearance (theme mode System, Light, or Dark, plus the sixteen-colour accent picker, all contrast-checked in both themes, added in the PART 8 update), Custom instructions (system-wide, issue #15), App lock (optional, off by default, described below). Data and connections: Web search (off by default, bring-your-own-endpoint: user pastes a SearXNG instance URL or a supported API key; when on, the chat can ground answers with results, clearly marked; when off the feature is invisible in chat), Backup and restore (hidden until built in Phase 7, not shown as coming text), Delete everything with a strong typed-word confirmation. Delete everything is data-only: it wipes chats, memory, projects, follow-ups, and Discover state, and leaves downloaded models in place (they are deleted individually in Storage), which is the safer default since re-downloading gigabytes is a real cost. A plain "Kam AI can be wrong" notice states in the app voice that the model is small and gets things wrong, to check what matters with the bookmark and Follow-ups, and that it is not a substitute for a professional. What Kam AI is for replays onboarding. Questions and answers with live-filtering search and the launch content from DESIGN.md section 10. About with the mark, version line, all links (YouTube @kamsiob, GitHub github.com/kamsiob, Website kamsiob.com, Telegram Kamsiob Lab invite, Feedback hello@kamsiob.com, Privacy policy, Licenses), the support line, and the Support this work button linking to buymeacoffee.com/kamsiob. The support button also closes the Settings page. Being considered and Not planned screen per DESIGN.md. Licenses screen listing all open source components and the Wikipedia CC BY-SA entry.

Report a response. On every AI response's overflow, a Report this response action that opens a prefilled email draft to hello@kamsiob.com containing the response text. Nothing sends unless the user sends it. This satisfies Play's AI-generated content reporting requirement; verify the current policy wording when preparing the listing and adjust if Google now requires a different mechanism.

Battery and thermal awareness. Monitor thermal status and battery during inference; degrade gracefully (shorter context, a plain warning) rather than cooking the phone, and say what is happening in plain words.

At-rest encryption and the optional app lock (PART 3, delivered on top of the working core). The whole database is encrypted at rest with SQLCipher, always on and transparent, keyed by 32 random bytes wrapped by an Android Keystore key (StrongBox-backed where available), with a real, verified migration of any existing plaintext database. On top of that, an optional app lock, off by default, gates the app itself: Device mode uses the phone's biometric or credential (recoverable, a UI gate over the encryption), and Passphrase mode adds a PBKDF2 layer over the wrapped key so the database cannot be opened at all without the passphrase (genuinely stronger, and unrecoverable if forgotten). The lock screen has a plain "Forgot your passphrase" path that erases the key and restarts rather than pretending to recover. The whole app is gated behind the lock at the top (KamRoot); the activity is a FragmentActivity so the biometric prompt can attach. The encrypted backup file, and gating of the other entry points (overlay, widget, tile, share, selection hook), integrate into their own phases.

Instrumented UI tests for onboarding completion, sending a message and receiving a streamed reply, bookmarking a response, and completing a follow-up. On-device manual pass on the Pixel over ADB following the walkthrough scripts defined at the end of this prompt, in both themes, plus an airplane-mode session proving full offline function, plus a process-death test (background the app, kill it, reopen, verify the conversation restored). ## Phase 5 · Discover and the Wikipedia pack pipeline

This phase has two halves. Build both in this order.

First, the pipeline, as a separate tools directory inside the repo. A config file maps branches of English Wikipedia's vital articles topic tree to packs. Launch packs: History, Science, How It Works (drawn from the technology and everyday-mechanisms branches), and People (biographies). Target 2,000 moments per pack; if a topic's configured branches genuinely cannot yield 2,000 entries that pass the quality filters, take everything they have and note the actual count in DECISIONS.md rather than padding with weak entries. The History pack must include, among its branches, civilizations and empires, important battles and wars, and major historical events and eras, and can draw more broadly across the history tree beyond those. The script walks the configured branches, pulls each article's introduction through the official Wikipedia API (use the plain-text extracts endpoint, set a proper descriptive User-Agent identifying the project and contact email per Wikimedia etiquette, respect rate limits with modest delays, and cache responses so reruns are cheap), cleans the text (strip reference artifacts, parenthetical pronunciation clutter, and layout junk), filters entries too short to make a good card, applies an exclude list file (article titles to skip, maintained reactively), and writes one versioned SQLite pack file per topic. Each entry stores: a stable id, title, topic, the cleaned intro text stored as both a generous preview (several paragraphs' worth, roughly 120 to 200 words, never a teaser) and the full passage, source article title and URL, license string CC BY-SA 4.0, and pack version. The script then computes a SHA-256 hash per pack file, writes or updates a manifest file (JSON) listing every pack with name, description, moment count, file size, version, download URL, and hash, and publishes the pack files and manifest as assets on a GitHub release in this repo using the gh CLI. GitHub release assets have no bandwidth limits and a 2 GiB per-file cap, so this scales to any user count at zero cost. Rerunning the script refreshes everything and bumps pack versions. Document one-command usage in the tools README. Run the pipeline now, generate all four launch packs, publish the release, and surface roughly twenty sample cards from the first pack in your output for the owner to skim once.

Second, the in-app Discover mode per DESIGN.md: the packs sheet reading the manifest, user-initiated downloads with progress and hash verification, the dealt card with its substantial preview per DESIGN.md (a real read of several paragraphs' worth, never a two-line teaser) and a bookmark to save it, the reader sheet with full passage and attribution and source link. The reader offers two clearly explained choices at the point of selection: Discuss this passage starts a grounded chat whose system instructions confine the model to the passage (it explains, discusses, and answers questions grounded in the text; when asked beyond it, it says so plainly and the bookmark catches it), and Explore this topic starts an ordinary open chat. A grounded discussion states its scope up front with a banner and offers a one-tap Continue in open chat that lifts the grounding, turns it into an open chat, and adds an honest note (issue #21). Then Deal another, local drawn-id tracking so only unseen cards are dealt with a plain reshuffle offer at true exhaustion, save and unsave on any card, the Saved section at the bottom of the Discover page, packs in Storage with delete, and the pack downloads described plainly in-app as the one-time network use they are. Saving is unified (issue #9): a saved moment is a follow-up carrying its pack and moment ids, so it appears in the one Follow-ups list under the Discover source filter and the Discover Saved section is a filtered view of that same list, not a separate store. The grounded discussion currently opens in the full chat window with the scope banner; a distinct scoped slide-up surface for it is intended but not yet built (open issue #11). Quiz me is a full flow per DESIGN.md: three to five questions generated strictly from the full passage, one at a time, answered by the user with plain per-question feedback that quotes what the passage actually said on a miss, a one-tap save to Follow-ups on any missed question, an end result like 4 of 5, and a quiet running tally overall and per pack (moments quizzed, questions right out of asked) shown as a small stats line in Discover and stored in the same database. Because the quiz draws on the full passage, if the user taps Quiz me without having opened the reader for the current card, first show a plain prompt asking whether they want to read the full moment before the quiz, with Read it first (opens the reader) and Quiz me anyway as the choices; once the reader has been opened for that card, start the quiz directly with no prompt. Explicitly no streaks, daily goals, XP, levels, badges, leaderboards, reminders, or pressure mechanics of any kind; the score exists only so a user who cares can see their own progress. Add the CC BY-SA entry to Licenses if not already present.

Testing gate: pipeline reruns idempotently; a corrupted download fails hash verification and recovers cleanly; deal-draw uniqueness verified by unit test over a small fixture pack; grounded-chat instructions verified by manual probes (ask beyond the passage, confirm the model declines and points to the bookmark); saved moments persist across process death; Discover works fully offline once packs are installed. ## Phase 7 · Files, export, and import

File attachments, conversation-level only in this version: plain text, Markdown, PDF with a text layer, and DOCX, with text extracted fully on-device. When a file exceeds the model's context, say so plainly and offer a page or section range instead of silently truncating. Explicitly excluded and listed under Not planned or Being considered as appropriate: images (needs a vision model per tier), spreadsheets (small models handle tables poorly), scanned PDFs (needs OCR), project-level files (waits for on-device embeddings).

Backup and restore: export writes everything (conversations, messages, memory, projects, follow-ups (which now include saved Discover moments, since saving is unified), Discover drawn state and quiz stats, settings) into one portable file the user saves wherever they choose; import restores it on any device, including migration to a new phone, with a clear merge-or-replace choice. The backup honours the app-lock passphrase model (encrypted backup, with the unrecoverable-passphrase warning and graceful wrong-passphrase handling). The codec is versioned and folds a legacy pre-unification "saved" array into follow-ups on import, so an older backup loses nothing. Downloaded models and packs are not embedded in the backup; after import, the app offers to re-download what is missing.

Testing gate: round-trip export-import equality verified by automated test on a populated database; oversized-file path verified with a large PDF; malformed-file handling; import of a backup from a differently-versioned schema handled with migration. Commit, push, refresh export.

## User testing protocol (used at every testing gate and in full at Phase 8)

Script A, the newcomer: fresh install. Complete onboarding reading everything, pick the recommended tier, let the download finish, ask three everyday questions, bookmark one answer, complete it in Follow-ups, find and read two Q and A entries by searching, flip the theme, close and reopen. Success means no confusion points, no visual defects, no dead ends.

Script B, the daily user: twenty-plus conversations seeded. Exercise all three chat list views, pin and unpin, archive, delete, collapse and expand Pinned, search for an old conversation, continue it, edit an earlier message and confirm truncation and re-answer, regenerate, stop mid-generation, use a project with instructions and confirm they hold across two chats, inspect and delete a memory entry.
