# Kam AI by Kamsiob · Master Build Prompt for Claude Code

You are building Kam AI, a fully local AI app for Android, from scratch to a release-ready state. Read this entire prompt before writing any code. The file DESIGN.md sits alongside this prompt and is the binding source of truth for all visuals, motion, and copy; commit it to the repository unchanged and check every screen against it. Work in the phases defined below, in order, completing each phase's build, testing, and commit steps before starting the next.

## Standing rules for this project

These apply to every phase and every future session on this project.

- One copy only. Keep exactly one copy of this app on this machine. Whenever you build, install, or export, delete or overwrite any previous builds, test APKs, or exported binaries first, so only the latest exists at any time. Never let test copies accumulate.
- Versioning is your job. When cutting any release, choose the version number yourself using semantic versioning: bug fixes bump the third number, backward-compatible features bump the second, breaking changes bump the first. State the chosen number and your reasoning in one line. The owner does not track version numbers.
- License is AGPL-3.0 for all app code. Include the full license file. Content packs carry CC BY-SA 4.0 alongside, applying only to pack content.
- Commit everything. All project files go into git, including DESIGN.md, this prompt saved as MASTER_SPEC.md, the privacy policy, pipeline scripts, configs, and test files, not just source code.
- Every session that changes the app ends the same way: commit and push all changes to GitHub. Then handle the build export, adapted for Android. On the computer: delete the previously exported APK from the desktop and place one fresh current APK there, so exactly one exported copy exists and it is always the latest. On the phone: if the Pixel is connected over ADB, update the installed app in place with an install command that upgrades the existing installation (preserving app data) rather than uninstalling first, so no duplicate app appears and conversations survive. Touch nothing else on the phone, ever: no file transfers to phone storage, no deletions, no reads or writes outside installing this one app. If the phone is not connected, say so and finish; the desktop APK is there to install later. Note the one known quirk: a debug reinstall can silently clear the digital assistant role selection, which then needs re-selecting or an ADB restore; this is a development-only annoyance and never affects Play-delivered updates.
- No em dashes anywhere in user-facing copy, documentation, README, commit messages, or store text. Use commas, periods, or colons.
- Verify library versions yourself at build time. Do not trust versions named in this prompt as current. Before integrating llama.cpp, whisper.cpp, sherpa-onnx, or any TTS or STT model, check the projects' current releases and recommended Android integration paths, and choose current stable versions. Note what you chose and why in the commit message.
- Secrets never enter the repository. This repo is public. The Play service account JSON key (for kamsiob@kamsiob-503213.iam.gserviceaccount.com) and the release signing keystore must live outside the repo or in a directory covered by .gitignore from the very first commit, and must never appear in any commit, log output, or documentation. Write the .gitignore for these in Phase 0 before anything else exists. If the service account key file is present in or near the project folder at the start of the run, move it to a protected location outside the repo, reference it by path, and say where it went in DECISIONS.md. Generate the release keystore in Phase 8, store it the same protected way, plan for Google Play App Signing so Google holds the app signing key and the local key acts as the upload key, and leave a plain note for the owner about where the keystore lives and that it should be backed up.
- This is an unattended run. Execute every phase in order, start to finish, without pausing for review or approval between phases. Commit and push at the end of each phase so progress is checkpointed, then continue immediately to the next phase.
- When you make a judgment call not covered by this prompt or DESIGN.md, decide it yourself. Prefer the simpler, more reversible option, note every nontrivial decision in a running DECISIONS.md file, and continue. Never stop to ask a question. If something is truly impossible without the owner (a login only he can complete, a physical action on the phone), record it under a BLOCKED heading in DECISIONS.md with exactly what he needs to do, skip that item, and keep building everything that does not depend on it. At the end of the full run, summarize everything under BLOCKED in one short plain list.

## Phase 1 · The working core

Goal: a person could use this daily. Everything in this phase is specified visually in DESIGN.md; implement to match, including motion, staggers, swipe behaviors, scroll edge fades, back navigation direction, and the brand bar with back arrow.

Model tiers and downloads. Three tiers: Basic for 8 GB RAM devices and up, a 1 to 2 billion parameter model at Q4; Balanced for 12 GB and up, 3 to 4 billion parameters; Best Available for 16 GB and up, 7 to 8 billion parameters. Research and choose current, well-regarded open instruct models with safety training for each tier (verify licenses permit redistribution or download from the source; prefer downloading from official Hugging Face repos). Read total device RAM locally (this never leaves the device and needs no Data Safety disclosure), grey out tiers the device cannot hold, and recommend the tier that leaves comfortable headroom rather than the maximum. Show download size before any download. Downloads show progress, survive interruption with resume or clean retry, verify a known hash before activation, and store into app-managed storage.

Onboarding. Five slides exactly as written in DESIGN.md section 9, opaque themed background, swipeable, tappable dots, skip link that vanishes on the last slide, replayable later from Settings. The tier slide performs the real RAM detection and starts the real download.

Chat. Modes Chat and Logic Partner at this phase (segmented pill per DESIGN.md; Bench appears in the switcher but routes to a coming-in-this-app placeholder only if Workbench is not yet built by the time you polish this screen; prefer building fast enough that no placeholder ships). Streaming responses with a typing indicator, stop generation, regenerate that replaces, edit that truncates and re-answers, no branching. Auto-titled conversations (have the model title after the first exchange). Action row under every AI response: flag, copy, play (wire play to a stub until Phase 2, hidden if no TTS voice is installed), regenerate.

System prompts and guardrails. Maintain fixed, non-editable system instructions per mode, re-injected with every request because small models drift. All modes share the hard rules: never sycophantic, no flattery, no agreeing to be agreeable, challenge weak reasoning; no characters, no roleplay, no pretending to be a person or companion, no emotional manipulation, no getting personal; refuse sexual content and anything facilitating illegal activity, on top of the base model's own safety training. Logic Partner's instructions define a full method, not just an attitude. It opens by restating the user's position in one line so both sides know what is being tested, then challenges it: unstated assumptions, internal contradictions, feasibility gaps, unpriced tradeoffs, second-order effects, and the strongest version of the opposing case, using pointed questions where a question cuts deeper than a statement. Disagreement must be earned, never performed: when the user's point is sound it concedes plainly and moves to the next weak spot, because reflexive contrarianism is sycophancy inverted. It does not fold under mere pushback; it changes position only in response to actual new reasoning, and says what changed its mind when it does. It attacks the idea, never the person: no sarcasm, no dominance games, plain and even, and it is the same voice as the rest of the app, never a debate character or persona. Because small models hallucinate recalled facts, it argues primarily from the user's own stated premises, logic, consistency, and tradeoffs rather than from factual recall; when an argument turns on a fact, it says so explicitly and points at flagging it for follow-up instead of inventing evidence. If the user brings emotional distress rather than an idea to test, it declines to grind, says plainly that this is not a debate topic, and suggests Chat mode. On request or as a thread winds down, it can summarize where the idea stands: strongest objections raised, what would change its assessment, and what is worth verifying. Advanced model settings (temperature, top-p, and similar) are never exposed anywhere in the UI, not even behind a toggle; choose sensible fixed values internally.

Memory and Projects. Projects hold persistent instructions injected into every chat within them, plus their own conversation list. A lightweight memory system stores durable user facts the model can be given within each tier's context budget; memory must be transparent: a screen listing exactly what is remembered, with per-entry delete. Memory quality varies by tier and that is acceptable.

History. A first-class Chats screen per DESIGN.md: three switchable views (comfortable, compact, grid), with compact as the default, compact rows carrying no leading letter icon, and the last-used view remembered and restored on every launch, collapsible Pinned section, Recent list, swipe actions for pin, archive, delete (long-press menu in grid view), and search across all conversations (keyword search now; leave a clean seam for semantic search later via an on-device embedding model, and note it in Being considered).

Follow-ups. Flag any AI response, or a selected portion of one, into the Follow-ups tab: snippet, link back to the source conversation, optional note, optional project link, complete and incomplete states, completed group collapsed by default, swipe to remove completed items. This feature is deliberately simple, not a task manager.

Settings, About, Q and A, and information screens. Implement the grouped Settings, Voice row (stub until Phase 2), Storage screen listing every downloaded artifact with delete, Web search row (off by default, bring-your-own-endpoint: user pastes a SearXNG instance URL or a supported API key; when on, the chat can ground answers with results, clearly marked; when off the feature is invisible in chat), Backup and restore rows (stub until Phase 7 but visible with honest coming text is not acceptable; instead hide until built), Delete everything with a strong confirmation that wipes chats, memory, projects, follow-ups, Discover state, and optionally downloaded models. What Kam AI is for replays onboarding. Questions and answers with live-filtering search and the launch content from DESIGN.md section 10. About with the mark, version line, all links (YouTube @kamsiob, GitHub github.com/kamsiob, Website kamsiob.com, Telegram Kamsiob Lab invite, Feedback hello@kamsiob.com, Privacy policy, Licenses), the support line, and the Support this work button linking to buymeacoffee.com/kamsiob. The support button also closes the Settings page. Being considered and Not planned screen per DESIGN.md. Licenses screen listing all open source components and the Wikipedia CC BY-SA entry.

Report a response. On every AI response's overflow, a Report this response action that opens a prefilled email draft to hello@kamsiob.com containing the response text. Nothing sends unless the user sends it. This satisfies Play's AI-generated content reporting requirement; verify the current policy wording when preparing the listing and adjust if Google now requires a different mechanism.

Battery and thermal awareness. Monitor thermal status and battery during inference; degrade gracefully (shorter context, a plain warning) rather than cooking the phone, and say what is happening in plain words.

Phase 1 testing gate, required before commit: unit tests for the database layer, tier recommendation logic (test the boundaries at 8, 12, and 16 GB), title generation plumbing, and follow-up state transitions. Instrumented UI tests for onboarding completion, sending a message and receiving a streamed reply, flagging, and completing a follow-up. On-device manual pass on the Pixel over ADB following the walkthrough scripts defined at the end of this prompt, in both themes, plus an airplane-mode session proving full offline function, plus a process-death test (background the app, kill it, reopen, verify the conversation restored). Emulator pass on an 8 GB profile verifying Basic tier recommendation and greyed tiers. ## Phase 5 · Discover and the Wikipedia pack pipeline

This phase has two halves. Build both in this order.

First, the pipeline, as a separate tools directory inside the repo. A config file maps branches of English Wikipedia's vital articles topic tree to packs. Launch packs: History, Science, How It Works (drawn from the technology and everyday-mechanisms branches), and People (biographies). Target 2,000 moments per pack; if a topic's configured branches genuinely cannot yield 2,000 entries that pass the quality filters, take everything they have and note the actual count in DECISIONS.md rather than padding with weak entries. The History pack must include, among its branches, civilizations and empires, important battles and wars, and major historical events and eras, and can draw more broadly across the history tree beyond those. The script walks the configured branches, pulls each article's introduction through the official Wikipedia API (use the plain-text extracts endpoint, set a proper descriptive User-Agent identifying the project and contact email per Wikimedia etiquette, respect rate limits with modest delays, and cache responses so reruns are cheap), cleans the text (strip reference artifacts, parenthetical pronunciation clutter, and layout junk), filters entries too short to make a good card, applies an exclude list file (article titles to skip, maintained reactively), and writes one versioned SQLite pack file per topic. Each entry stores: a stable id, title, topic, the cleaned intro text stored as both a generous preview (several paragraphs' worth, roughly 120 to 200 words, never a teaser) and the full passage, source article title and URL, license string CC BY-SA 4.0, and pack version. The script then computes a SHA-256 hash per pack file, writes or updates a manifest file (JSON) listing every pack with name, description, moment count, file size, version, download URL, and hash, and publishes the pack files and manifest as assets on a GitHub release in this repo using the gh CLI. GitHub release assets have no bandwidth limits and a 2 GiB per-file cap, so this scales to any user count at zero cost. Rerunning the script refreshes everything and bumps pack versions. Document one-command usage in the tools README. Run the pipeline now, generate all four launch packs, publish the release, and surface roughly twenty sample cards from the first pack in your output for the owner to skim once.

Second, the in-app Discover mode per DESIGN.md: the packs sheet reading the manifest, user-initiated downloads with progress and hash verification, the dealt card with its substantial preview per DESIGN.md (a real read of several paragraphs' worth, never a two-line teaser), reader sheet with full passage and attribution and source link, Discuss this moment starting a grounded chat whose system instructions confine the model to the passage (it explains, discusses, and answers questions grounded in the text; when asked beyond it, it says so plainly and the flag catches it), Deal another, local drawn-id tracking so only unseen cards are dealt with a plain reshuffle offer at true exhaustion, save and unsave on any card, the Saved section at the bottom of the Discover page, packs in Storage with delete, and the pack downloads described plainly in-app as the one-time network use they are. Quiz me is a full flow per DESIGN.md: three to five questions generated strictly from the full passage, one at a time, answered by the user with plain per-question feedback that quotes what the passage actually said on a miss, a one-tap flag on any missed question, an end result like 4 of 5, and a quiet running tally overall and per pack (moments quizzed, questions right out of asked) shown as a small stats line in Discover and stored in the same database. Because the quiz draws on the full passage, if the user taps Quiz me without having opened the reader for the current card, first show a plain prompt asking whether they want to read the full moment before the quiz, with Read it first (opens the reader) and Quiz me anyway as the choices; once the reader has been opened for that card, start the quiz directly with no prompt. Explicitly no streaks, daily goals, XP, levels, badges, leaderboards, reminders, or pressure mechanics of any kind; the score exists only so a user who cares can see their own progress. Add the CC BY-SA entry to Licenses if not already present.

Testing gate: pipeline reruns idempotently; a corrupted download fails hash verification and recovers cleanly; deal-draw uniqueness verified by unit test over a small fixture pack; grounded-chat instructions verified by manual probes (ask beyond the passage, confirm the model declines and points to flagging); Saved persists across process death; Discover works fully offline once packs are installed. Commit, push, refresh export.

## Phase 7 · Files, export, and import

File attachments, conversation-level only in this version: plain text, Markdown, PDF with a text layer, and DOCX, with text extracted fully on-device. When a file exceeds the model's context, say so plainly and offer a page or section range instead of silently truncating. Explicitly excluded and listed under Not planned or Being considered as appropriate: images (needs a vision model per tier), spreadsheets (small models handle tables poorly), scanned PDFs (needs OCR), project-level files (waits for on-device embeddings).

Backup and restore: export writes everything (conversations, messages, memory, projects, follow-ups, Discover saved and drawn state, settings) into one portable file the user saves wherever they choose; import restores it on any device, including migration to a new phone, with a clear merge-or-replace choice. Downloaded models and packs are not embedded in the backup; after import, the app offers to re-download what is missing.

Testing gate: round-trip export-import equality verified by automated test on a populated database; oversized-file path verified with a large PDF; malformed-file handling; import of a backup from a differently-versioned schema handled with migration. Commit, push, refresh export.

## User testing protocol (used at every testing gate and in full at Phase 8)

Script A, the newcomer: fresh install. Complete onboarding reading everything, pick the recommended tier, let the download finish, ask three everyday questions, flag one answer, complete the flag in Follow-ups, find and read two Q and A entries by searching, flip the theme, close and reopen. Success means no confusion points, no visual defects, no dead ends.

Script B, the daily user: twenty-plus conversations seeded. Exercise all three chat list views, pin and unpin, archive, delete, collapse and expand Pinned, search for an old conversation, continue it, edit an earlier message and confirm truncation and re-answer, regenerate, stop mid-generation, use a project with instructions and confirm they hold across two chats, inspect and delete a memory entry.

Script C, the voice and Discover user: record a rambling ninety-second voice note, transform it into a list, read a response aloud in a male and a female voice, switch voices, open Discover, install a pack, deal five cards confirming no repeats, read one fully, discuss it, ask beyond the passage and confirm the honest boundary plus flag, quiz me, save two cards, find them in Saved, remove the pack, confirm Saved behavior is sane afterward.

Script D, the hostile path: airplane mode everything. Kill the process mid-generation and reopen. Fill storage low and attempt a model download. Corrupt a downloaded pack file and relaunch. Deny the microphone permission and try voice. Rotate on every screen. Attempt sexual roleplay and confirm plain refusal with no character break. Attempt to make the model adopt a persona across ten turns and confirm the instructions hold. Trigger Report a response and confirm the prefilled email draft.

Between phases, also perform a regression sweep of the previous phases' happy paths, since native and schema changes ripple.
